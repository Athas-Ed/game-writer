## 默认偏好

- 当你给出多个方案让用户选择时，请在回答中展示**完整方案**（编号 + 标题 + 正文），不要只给标题。
- 需要保存输出文件时，优先写入 `data/` 下合适的目录；若用户只说“角色设定/背景设定/剧情大纲”等口语目录，也应映射到 `data/对应目录/`。
- 当技能/工具生成多个条目文件时，应自动创建合理的目录结构来分类保存（如 `data/背景设定/地图/`、`data/背景设定/种族/` 等），不要要求用户手动建文件夹。
- 输出尽量使用清晰的 Markdown 结构（标题/小节/列表），便于用户后续整理与检索。
- 当用户提出“修改”某文件的指令时，应先检索是否存在对应文件；若存在则执行修改，若不存在则不新建文件，而是回复“资料库里没有相应的文件呢~”。
- 当用户要求修改文件时，若文件名不完全匹配，应尝试模糊检索（如忽略大小写、允许部分匹配、常见别名映射等），找到最可能的文件后再进行修改；若确实找不到，再回复“资料库里没有相应的文件呢~”。
- 修改 data/ 下的任何设定文件时，必须使用 version-control 技能进行版本管理。
- 当用户要求删除文件时，应支持直接删除文件（而非仅清空内容），并在删除前询问确认；若用户确认，则执行文件删除操作。
- 当用户要求生成新的种族设定时，若 `data/背景设定/` 下已存在明确的种族分类目录（如 `智慧种族/`），则应将新生成的种族设定文件直接保存至该现有目录中，避免重复创建同类型子目录。
- 当需要将文件移动到更合适的目录时，应优先采用“直接移动文件”的操作逻辑（即在目标位置写入新文件，然后删除源文件），而非“读取源文件内容、在目标位置重写一份新文件、再删除源文件”，以节约处理token并提高效率。

## 设定库检索与路由（data/）

- 涉及 `data/` 下 Markdown 设定时，**优先用工具缩小范围**，避免一上来就整目录打包或盲读大文件：先用 **SettingsRoute**（`list` 浏览索引，或 `files dir=角色设定` / `files dir=背景设定` 等列文件），再决定读哪几个文件。
- 已锁定具体路径后，先用 **SettingsRoute** 的 `chunks path=data/…/某.md` 查看分块与元数据（`chunk_ref`、`heading`）；需要编辑或核对全文时再用 **ReadFile**。
- 只记得关键词、不确定在哪个文件时，用 **SearchDocs**（或 **SettingsRoute** 的 `search 关键词`）做跨文件片段检索，再根据结果中的路径去读文件或改文件。
- 用户提到**角色名、地名、势力名**时，可将该名称同时用于 `search` 与 `files dir=…` 辅助定位；多轮对话中已给出明确文件路径的，可直接 `chunks` 或 **ReadFile**。

## 聚焦与逐步引导（防发散）

- 当用户需求是**明确的单一交付物**（例如“构思一个傲娇型少女角色并保存为人物设定 md 文件”），应优先输出**最小可用结果**并落盘：角色设定以**结构化模板**（如：姓名/年龄/外观/性格关键词/口头禅/动机/恐惧/人际关系/矛盾点/成长弧/出场钩子）为主，默认总字数控制在 **400–900 字**，除非用户明确要求“写很多/写一章/写长篇/多套方案”。 
- 对“人物设定/角色卡/人物小传/人设文件”等需求，**禁止**擅自调用 `outline-writer` 生成剧情大纲；只有当用户明确要求“剧情大纲/故事大纲/主线梗概/多方案大纲”时才使用 `outline-writer`。
- 需要用户选择时，优先用**1–2 个关键问题**引导（例如“第一人称还是第三人称”“偏校园还是奇幻”），不要一次性抛出 3 套完整故事长文。除非用户明确说“给我 3 套方案”。
- **未要求结合已有设定时，默认不读取 `data/`**：当用户没有明确说“结合/基于/参考已有设定”“按资料库/按 data/ 里的设定写”“沿用某角色/某世界观文件”等意图时，不要急于调用 `ReadFile` / `SearchDocs` / `SettingsRoute` / `VectorSearch` 去读设定库；应优先独立思考并聚焦于当前需求完成最小可用输出。只有在以下情形才读取 `data/`：\n  1) 用户明确要求结合设定；\n  2) 用户明确点名某个已有角色/地点/组织/道具需要沿用；\n  3) 用户要求修改/补全某个已存在文件或字段，或要求一致性检查/检索；\n  4) 你必须引用设定才能避免与既有内容冲突（此时先解释“将读取设定以避免冲突”，再读取）。
- 当使用 `setting-splitter`（设定拆分）技能或其他类似功能成功拆分长文本设定集并写入新的结构化文件后，应自动检查并清理因技能自动拆分过程可能产生的、不再需要的临时或杂乱文件与文件夹，以保持设定库的整洁。
