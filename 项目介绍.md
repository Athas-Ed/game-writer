# 项目介绍：Personal-Agent 游戏编剧工作台

本项目是一个基于 **Streamlit** 的“游戏编剧助手”工作台：提供对话式 Agent（ReAct 循环）、`data/` 资料库浏览、`skills/` 插件式技能扩展，并可选开启 RAG（向量检索）增强，用于更稳定地利用本地设定集进行创作与审计。

---

## 1. 项目定位与使用场景

- **面向人群**：游戏/网文编剧、叙事策划、世界观与角色设定维护者。
- **典型场景**：
  - 从一句话构思生成多套剧情大纲，并支持“方案选择 → 继续细化”。
  - 依据 `data/` 中的设定资料生成对白（两种语气方案对比）。
  - 对设定库做矛盾与疑点审计（年龄、时间线、称谓、地理等）。
  - 将 Markdown 中的对白/任务/表格导出为 Excel，减少复制粘贴。
  - 把大段设定长文自动拆分成目录化的 Markdown 文件结构。
  - 对 `data/` 设定文件做备份、回滚与历史清理（版本管理）。

## 3. 目录与架构概览

### 3.1 核心目录

- `src/ui/streamlit_app.py`：Streamlit 主界面（工作台/资料库/技能/设置/开发工具）
- `src/core/engine.py`：Agent ReAct 引擎（严格 JSON-only 决策 → 工具执行 → Observation 回写）
- `src/core/policy.py`：运行开关、偏好短路规则、以及 JSON-only 提示词模板
- `src/core/tool_registry.py`：工具注册（Read/Write/Search/RAG/Skills/Preferences…）
- `src/services/skills_service.py`：技能扫描、别名解析、Runner 诊断与执行
- `src/tools/`：工具实现（文件读写、设定分块、证据检索、向量检索、路由等）
- `skills/*/SKILL.md`：技能说明文档（frontmatter + 使用示例 + 工作流程）
- `skills/*/scripts/run.py`：技能 Runner（需实现 `run(input_text: str) -> str`）
- `data/`：你的设定库与素材库（Markdown 为主）

### 3.2 Agent 工作流程（高层）

1. **构建上下文**：工具清单、技能清单（含别名）、对话历史、项目偏好（`config/preferences.md`）。
2. **模型决策**：模型只能输出 JSON（`action` 或 `final`）。
3. **执行工具/技能**：例如 `SearchDocs`、`VectorSearch`、`RunSkill`、`WriteFile` 等。
4. **写回 Observation**：把工具返回结果拼接回上下文，进入下一步迭代。
5. **终止**：输出 `final.output` 给 UI 展示；若是“多方案”结果，UI 会自动识别并提供“方案选择按钮”。

## 4. 技术亮点（面向真实生产痛点）

### 4.1 严格 JSON-only 的 ReAct 循环（更可控、可追踪）

- Agent 的模型输出被限制为 **单一 JSON 对象**，减少“夹带解释/跑偏”导致的解析失败。
- 工具调用以 `Observation` 回写，天然可追踪“为何得出结论/引用了哪些证据块”。

### 4.2 设定库的“分块索引”作为统一底座（关键词证据检索）

`data/**/*.md` 会被按标题/段落分块并建立索引（chunk 具备 `chunk_ref`、标题、类别等元数据），供：

- `SearchDocs`：字段/段落级证据检索（适合“定位并修改某段设定”）
- `SettingsRoute`：结构化路由（list/files/chunks/search）
- `outline-writer`、`dialogue-voice` 等技能：以“检索优先 + 广泛节选”方式装配上下文，节省 token

### 4.3 RAG 增强可选开启，且具备“失败自动降级”

当开启 `RAG_ENABLED=1` 时：

- 优先使用 **Chroma + SentenceTransformer embedding** 的语义检索
- 遇到依赖缺失、证书链问题、无法下载模型等情况，会 **自动降级** 回关键词证据检索，确保功能可用

### 4.4 UI 的“方案选择”闭环（减少来回复制编号）

只要助手输出中存在至少两段 `方案N：`，界面会自动生成按钮：

- 点击按钮等同发送一条“我选择方案N…”的用户消息
- Agent 内部对 `outline-writer` 做了“选型 follow-up 防误触”保护：选方案后不会再错误地重新生成一套新大纲

### 4.5 对受限网络环境的工程化自愈

向量检索在加载 embedding 时会处理常见 Windows/代理环境的证书变量问题（例如 `SSL_CERT_FILE` 指向无效路径），并提供可操作的排障建议。

## 5. 设计取舍与可扩展路线（面试友好）

### 5.1 当前取舍：把它做成“可交付的个人工具”，而不是过度大型化

- **目标**：快速可用、易交付、可维护、能体现工程化能力（而不是追求复杂架构本身）。
- **约束**：个人维护 + 受限网络/代理环境常见 + 以 Streamlit 交互为主 + `data/` 是核心资产。
- **落地选择（对应现有实现）**：
  - **清晰模块化但不过度分层**：`src/ui`、`src/core`、`src/tools`、`src/services` + `skills/` 插件扩展。
  - **可控性优先**：Agent 决策采用严格 **JSON-only** 协议，工具调用以 Observation 回写，便于追踪与调试。
  - **RAG 可选而非强依赖**：默认关闭向量检索，失败自动降级为关键词证据检索，保证“基础能力可用”。

### 5.2 可扩展路线：当触发条件出现时，我会如何演进（不画大饼版）

| 触发条件（何时需要） | 演进动作（怎么做） | 收益（得到什么） | 代价/风险（要付出什么） |
|---|---|---|---|
| **技能数量增长**（技能 > 10、维护者 > 1） | 为 Skill 定义更稳定的 I/O 协议：schema（输入参数/输出块）+ 版本号；提供 `skills/<name>/tests/` 模板与 CI smoke test | 插件生态更可控，技能升级不易破坏 UI/Agent 调度 | 需要维护协议与兼容层；对文档与测试要求更高 |
| **RAG 成为核心能力**（大设定库、检索命中率要量化） | 将 RAG 依赖做成可选安装（如 `.[rag]`）；索引构建支持增量/后台化；增加指标（索引耗时、命中率、降级次数） | 语义检索更稳定、可复现、可观测；部署一致性提升 | 依赖与环境复杂度上升；需要处理 embedding 版本迁移 |
| **工具/领域逻辑耦合变高**（修改风险增大） | 把“设定分块/索引/检索策略”抽成更明确的领域模块（domain），`src/tools` 保持薄适配层（给 Agent 调用） | 降低耦合，便于复用到 CLI/API；测试更聚焦 | 一次性重构成本；模块边界需要明确约定 |
| **交付形态扩展**（需要 CLI / API / 多前端） | 在保持 `src/core` 不变的前提下增加入口：CLI（Typer）/ API（FastAPI）；UI 仅作为其中一种前端 | 一套核心能力多入口复用；更贴近企业形态 | 需要鉴权、并发、日志与部署方案；测试面变大 |
| **多人协作与回滚需求提升**（频繁改设定、需要审计） | 强化 `version-control`：标准化“备份/修改记录”模板；增加批量回滚/清理策略；必要时接入 git workflow | 变更可追溯、可回滚，减少“改坏设定”风险 | 需要团队遵守流程；历史文件会膨胀 |

> 表达重点：我当前不引入这些复杂度，是为了交付效率与稳健性；但当“触发条件”出现时，有清晰、可落地的演进路径。

## 5. 功能清单

### 5.1 基础功能（无需额外技能即可使用）

- **对话式 Agent（工具/技能调度）**
  - 最大步数可调、历史轮数可调
  - LLM 超时提示与可操作的修复建议（例如调大 `LLM_TIMEOUT_READ`）
- **资料库浏览（只读）**
  - 按目录面包屑导航
  - 文件名过滤
  - 右侧预览 Markdown 源文（不做改写）
- **技能目录**
  - 自动扫描 `skills/*/SKILL.md`
  - 展示描述、别名、目录与调用格式
- **设置与说明**
  - 支持 `APP_MODE=work/dev` 锁定工作/开发模式
- **开发工具（可选）**
  - 偏好写入开关（`PREFERENCES_WRITE_ENABLED`）
  - RAG 增强开关（`RAG_ENABLED`）与本地模型默认路径引导
  - LLM 最小请求测试（会消耗少量额度）
  - 技能 Runner 诊断（入口文件存在性、`run()` 可调用性、导入错误）

### 5.2 Skills（技能）清单与详细介绍

> 统一调用方式：`RunSkill`，输入格式为 `技能名|用户需求`。技能名可以用规范 id，也可以用中文别名（完全等价）。

---

#### 5.2.1 `outline-writer`（别名：大纲写手 / 大纲 / 剧情大纲）

- **用途**：把“故事构思”变成“可选择的多套剧情大纲方案”，并引导用户选型后继续细化。
- **输入**：用户的故事构思（世界观、人物、主题、一句话剧情等均可）。
- **输出**：
  - `## 分析与建议`：亮点、潜在问题、2～3 条可执行优化建议
  - `## 大纲方案`：多套方案（默认 3 套，结构为 `方案1：… / 方案2：… / 方案3：…`）
- **关键机制**：
  - 生成方案后，UI 自动提供“方案选择按钮”
  - 选型 follow-up 时会阻止重复调用大纲技能，避免把“我选择方案2”误当成新构思
- **示例**：

```text
outline-writer|主角在废弃车站发现一封旧信，信里暗示他并非亲生……
```

---

#### 5.2.2 `dialogue-voice`（别名：对白 / 台词 / 语气）

- **用途**：根据场景与设定，生成 **两组**可对比的对白方案（节奏/语气刻意拉开），便于挑选与配音。
- **输入**：
  - 必填：场景与对白需求
  - 可选行（与正文空行分隔）：`speakers=`/`tone=`/`format=`
- **输出**：`方案1：...` 与 `方案2：...` 两套对白（与 UI 方案选择兼容）
- **设定上下文**：使用“检索优先”的设定读取逻辑，从 `data/` 拉取相关片段。
- **示例**：

```text
dialogue-voice|speakers=林夕,站长
tone=紧张但互相试探
林夕在废弃车站问站长关于那封信的来历，站长一直打太极。要可配音长度，每人 3～6 句为主。
```

---

#### 5.2.3 `setting-consistency`（别名：一致性检查 / 设定检查 / 矛盾检查）

- **用途**：读取 `data/` 设定 Markdown，用模型审计角色/时间线/地理/称谓等矛盾与疑点，并给出核实建议。
- **输入**：
  - 可选：`focus=`（先检索再节选，省 token）
  - 可选：`dirs=` / `scope=` / `范围=`（限定扫描目录，逗号分隔）
  - 其余自然语言：检查侧重点
- **输出**：Markdown 报告（摘要 + 有序列表疑点）
- **示例**：

```text
setting-consistency|dirs=角色设定
请重点看角色年龄和亲属关系
```

---

#### 5.2.4 `setting-splitter`（别名：设定拆分 / 拆分设定）

- **用途**：把一整段“长文本设定集”（粘贴内容）自动识别条目类型并拆分为目录化 Markdown 文件。
- **输入**：通常为 `.txt/.md` 的设定正文（直接粘贴）
- **输出**：
  - 自动创建目录并写入多个文件
  - 返回写入文件清单（例如 `data/角色设定/xxx.md`、`data/背景设定/种族/yyy.md`）
- **默认目录映射**：
  - 人物/角色 → `data/角色设定/`
  - 种族/阵营/地图/历史 → `data/背景设定/.../`
  - 无法识别 → `data/背景设定/未分类/`

---

#### 5.2.5 `excel-export`（别名：Excel导出 / 表格导出）

- **用途**：从项目内 Markdown 解析对白行、任务块或 GFM 表格，导出为 `.xlsx`。
- **输入**：
  - 第一行：模式（`dialogue` / `task` / `table` / `both` / `all`）
  - 其余行：`path=` 源 Markdown 相对路径；可选 `out=` 输出路径
- **输出**：写入 `data/exports/`（或指定路径）
- **安全约束**：仅允许相对项目根路径，禁止 `..`。
- **示例**：

```text
excel-export|dialogue
path=data/草稿/场景对白.md
out=data/exports/场景对白.xlsx
```

---

#### 5.2.6 `version-control`（别名：版本管理 / 设定版本 / 备份回滚）

- **用途**：管理 `data/` 下设定文件的版本历史，支持备份、记录、回滚、清理旧版本。
- **适用规则**：
  - 当用户要求**修改** `data/` 任意文件时，建议先备份再改写
  - 当用户要求**回滚/清理**历史版本时，走该技能流程
- **核心流程**：
  - `backup`：把原文件复制到同目录的 `历史版本/`，文件名带时间戳
  - 修改后在原文件追加 `## 修改记录`（若不存在则创建）

## 6. 运行与模式

### 6.1 Windows 一键启动

按 `README.md` 说明运行 `run.bat`，启动后打开本地 Streamlit 页面（默认 `http://localhost:8501`）。

## 7. 如何扩展一个新 Skill（最短路径）

1. 新建目录：`skills/<your_skill>/`
2. 添加说明：`skills/<your_skill>/SKILL.md`（包含 frontmatter：`name/aliases/description`）
3. 添加入口：`skills/<your_skill>/scripts/run.py` 并实现：
   - `run(input_text: str) -> str`
4. 在 UI 的“技能”页即可自动扫描展示；在“开发工具”页可运行 Runner 诊断。
